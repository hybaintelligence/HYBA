================================================================================
ARCHITECTURE SUMMARY
================================================================================

The wiring creates a 4-stage initialization pipeline:

Stage 1 (App Startup):
  → DistributedLockManager created in main.py lifespan
  → Stored in app.state for access by all handlers
  → Redis connection status logged

Stage 2 (Engine Creation):
  → Lock manager retrieved from globals or passed explicitly
  → Passed to UnifiedMiningEngine constructor
  → Stored as engine.lock_manager

Stage 3 (Controller Creation):
  → Lock manager passed to AutonomousMiningController
  → _initialize_hardening_modules() called
  → All 5 modules instantiated (gracefully handle failures)

Stage 4 (Runtime Use):
  → ReflexiveCycleGuard wraps reflexive cycle phases
  → Other modules available for mining path integration
  → Metrics exported for monitoring

This architecture ensures:
  1. Single lock manager per app instance
  2. Modules initialized once at startup (not per-request)
  3. Graceful degradation if Redis unavailable
  4. Full observability via Prometheus metrics
  5. Production-safe defaults (AUTO_DENY, 100ms timeout, etc.)
