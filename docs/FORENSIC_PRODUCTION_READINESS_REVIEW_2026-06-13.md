# HYBA_FULLSTACK Forensic Production Readiness Review — 2026-06-13

Scope: HYBA_FULLSTACK as the separated funding-engine repository

## Decision

HYBA_FULLSTACK is certified production-ready for a controlled command-room deployment path, provided the local evidence packets generated by the operator are preserved with the release record.

This certification covers:

- production-mode local/runtime deployment;
- bridge and backend readiness checks;
- operator-authenticated mining control;
- deterministic PYTHIA search repeatability gate;
- Phi^15 empirical evidence artefact validation gate;
- MIDAS state-machine protected pool connection;
- controlled live-pool accepted-share capture.

This certification does **not** by itself certify:

- sustained profitability;
- payroll reliance;
- office-cost reliance;
- stable hashrate;
- unattended autonomous mining;
- first seven MD offer release before accepted-share evidence;
- any revenue claim without local and pool-side accepted-share proof.

## Forensic findings

### 1. Repository separation is correct

HYBA_FULLSTACK is the separate funding-engine repository. It is intentionally separated from HYBA_Unified_Backend / HYBA_Unified_Frontend so treasury-supporting mining, deployment evidence, and accepted-share gates are governed as their own operational rail.

This is correct. Funding operations should remain separate from the broader METIS / EUCLID institutional backend.

### 2. Production readiness path now matches actual operating reality

The production readiness runbook now recognises two valid evidence paths:

1. hosted CI; or
2. local command-room evidence when CI cannot be completed.

The local path requires:

```bash
npm ci
npm run test:funding:gate
npm run funding:gate
npm run prod:local:gate
npm run prod:live:gate
npm run build
```

This aligns the repository with the current runway reality: production readiness can be based on local checks, provided JSON evidence packets are preserved with commit SHA, operator, timestamp, terminal output, and reviewer sign-off.

### 3. Local production gate is evidence-grade

`scripts/local_production_gate.py` runs the production evidence categories locally, captures stdout/stderr tails, records host and environment summary, and writes a timestamped JSON packet under `artifacts/production_readiness/`.

It does not connect to pools and does not enable live share submission. That is correct separation: readiness first, operator-controlled mining later.

### 4. Funding-engine deployment gate is appropriate

`scripts/funding_engine_deployment_gate.py` validates three funding-engine preconditions:

1. Phi^15 empirical evidence CSV/JSON artefacts are present and schema-valid.
2. JSON summary values agree with CSV-derived resonance metrics.
3. PYTHIA deterministic search returns the same nonce for the same target/range.

In accepted-share mode it also requires local accepted-share evidence before the MD-offer trigger is satisfied.

This is the right gate design: empirical evidence, deterministic search, then accepted-share proof.

### 5. Deterministic search is explicitly tested

The funding gate runs PYTHIA twice against the same target and nonce range. The test suite asserts that both runs return the same nonce and that the nonce is inside the requested search range.

This is sufficient to certify local deterministic repeatability for the deployment gate. It is not a claim that every pool will accept the returned nonce; accepted-share proof remains the funding threshold event.

### 6. MIDAS mining controls are production-grade enough for controlled deployment

The mining API enforces role-gated read/control access, request IDs, idempotency keys, rate limiting, backpressure control, MIDAS state transitions, and a governed 1 EH/s PULVINI capacity cap.

The mining lifecycle remains explicitly operator controlled. Auto-connect is not the default launch posture.

### 7. Mining health issue previously identified is resolved

The mining health endpoint now checks `midas_validation["valid"]` directly. This means the health endpoint no longer treats a non-empty validation dictionary as automatically healthy.

This removes a previous production-readiness concern.

### 8. Command-room mining session runner is now safe and reproducible

`start_mining_session.py` no longer embeds operator or pool credentials. It requires credentials from environment variables and captures:

- bridge health before session;
- backend readiness before session;
- operator login summary;
- true pre-config mining status;
- pool config response;
- post-config/pre-connect status;
- pool connect response;
- post-connect status;
- per-minute mining status;
- final pre-disconnect status;
- pools status;
- disconnect response;
- post-disconnect status.

This is the right evidence chain for accepted-share capture.

### 9. Dependency reproducibility issue fixed during review

The command-room runner imports `requests`. During review, `requests` was present in the script dependency path but missing from the Python requirement locks. This has been fixed in both:

- `python_backend/requirements.txt`;
- `python_backend/requirements.phase-transition.txt`.

This prevents a clean command-room machine or container environment from failing due to a missing evidence-runner dependency.

### 10. Evidence packet preservation remains mandatory

`artifacts/production_readiness/` and `artifacts/funding_engine/` are intentionally ignored by Git because they may contain runtime evidence and local deployment metadata.

That is correct, but it means production certification depends on preserving those packets in the command-room release folder, not assuming they are in repository history.

The operator must preserve:

- local production gate JSON;
- live gate JSON;
- funding gate JSON;
- terminal output;
- redacted environment summary;
- command-room mining-session folder;
- pool-side accepted/rejected share export or screenshot;
- approval ticket.

## Remaining non-blocking controls before first accepted-share run

These are not blockers to deployment, but they are mandatory operating controls:

1. Use a sealed local environment or secret manager for all JWT, operator, and pool credentials.
2. Keep `HYBA_ENABLE_MINING_AUTOCONNECT=false` at startup.
3. Keep `HYBA_ENABLE_LIVE_SHARE_SUBMIT=false` until explicit approval is attached.
4. Start mining only through authenticated operator/MIDAS action.
5. Preserve all local evidence packets outside Git history.
6. Capture pool-side accepted/rejected share evidence.
7. Run `npm run funding:accepted-share:gate` after the accepted share appears locally.
8. Release the first seven MD offers only after the approval ticket confirms accepted-share evidence.
