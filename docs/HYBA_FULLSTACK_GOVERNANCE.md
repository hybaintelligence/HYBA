# HYBA_FULLSTACK Governance Charter

Status: Canonical operating-governance note  
Owner: HYBA Group / Company  
Scope: HYBA_FULLSTACK only  
Last updated: 2026-06-12

## BLUF

HYBA emerged from METIS. METIS is the root substrate: one repo, 10,000 governed uses.

HYBA_FULLSTACK is HYBA's self-financing operating substrate. It exists to generate treasury capacity for HYBA Group, sustain production operations, and support the wider humanitarian mission without forcing HYBA Foundation or HYBA Research to become revenue engines.

It is not the whole HYBA portfolio. It is not the Foundation. It is not HYBA Research. It is not THEMIS. It is a separate production repo and self-financing expression of the METIS doctrine: the production-grade full-stack runtime for the PYTHIA/PYTHAGORAS mining path, the operator console, the FastAPI backend, the Stratum protocol boundary, and the PULVINI memory-compression/manifold layer used by this operating system.

## METIS relationship

HYBA_FULLSTACK does not replace METIS and does not create a separate institutional origin story.

METIS is the root substrate. HYBA_FULLSTACK is one production use of that substrate: a treasury-supporting runtime that helps fund the wider HYBA mission.

The relationship is:

```text
METIS -> governed substrate
HYBA_FULLSTACK -> self-financing production runtime emerging from the substrate
MIDAS / PYTHIA / PYTHAGORAS -> mining and mathematical execution pathway
PULVINI -> memory compression at scale inside the runtime
THEMIS -> governance and evidence authority
```

## Self-financing mandate

HYBA_FULLSTACK is the financial independence layer of the institution.

The mandate is to:

1. Generate lawful treasury capacity from production-grade runtime activity.
2. Fund the operating company without compromising the Foundation's public-good independence.
3. Support long-horizon research without forcing HYBA Research into short-term commercial delivery.
4. Preserve auditability, operator control, and anti-simulation discipline so revenue is evidence-bound.
5. Keep mining operations separate from treasury allocation, custody, payroll, tax, creditor, regulatory, and solvency decisions.

The correct phrase is:

```text
METIS is the root. HYBA_FULLSTACK funds the mission. HYBA Foundation protects the mission. HYBA Research advances the frontier. THEMIS governs evidence and permissioning. PULVINI compresses memory at scale.
```

## Operating governance rules

HYBA_FULLSTACK follows these rules:

1. Production mining paths must use real operator configuration, real pool messages, real hash/share outcomes, and deterministic mathematical transforms.
2. Development fixtures must be isolated behind explicit gates.
3. Production must reject fabricated mining values, pseudo-random runtime telemetry, and simulated target-job injection.
4. Operator credentials must remain secret-managed and use production-grade hashing.
5. Live share submission must remain disabled until explicit launch approval exists.
6. Every mutating mining operation must pass through MIDAS controls, request tracking, rate limiting, and backpressure guards.
7. Mining runtime controls must remain separate from treasury allocation and custody decisions.
8. Scientific or performance claims must be scoped to the evidence currently present in certificates, tests, and live pool telemetry.

## Claim boundaries

HYBA_FULLSTACK may claim, where evidenced:

- deterministic protocol handling;
- anti-simulation production guardrails;
- mathematical certificate generation;
- PULVINI memory-compression and state-surface behaviour;
- operator-controlled mining runtime readiness;
- local proof-of-work validation;
- Stratum protocol integration readiness;
- production-readiness gate structure.

HYBA_FULLSTACK must not claim, unless independently measured and legally approved:

- guaranteed mining revenue;
- pool-side hashrate from local mathematical filters alone;
- quantum speedup over SHA-256;
- accepted shares without real pool confirmation;
- regulatory approval;
- treasury solvency;
- Foundation impact without Foundation measurement;
- scientific breakthrough without HYBA Research approval.

## Evidence hierarchy

Use this evidence hierarchy for internal reviews and external packs:

1. Live pool telemetry and accepted/rejected share logs.
2. Production readiness gates, local command-room evidence packets, and CI results where available.
3. Runtime anti-simulation guard outputs.
4. Local proof-of-work validation records.
5. PULVINI mathematical certificates.
6. Property, unit, integration, and E2E tests.
7. Documentation notes and explanatory material.

Claims must be made at or below the strength of the available evidence.

## Release governance checklist

Before tagging a release candidate, the accountable operator must confirm:

- production environment validation passed;
- runtime anti-simulation guard passed;
- backend regression tests passed;
- frontend build and typecheck passed;
- Docker image built;
- local command-room evidence packet retained where CI is unavailable;
- live share submission posture is explicitly recorded;
- operator credentials are secret-managed and hashed;
- pool credentials are real, operator-owned, and not committed;
- treasury allocation authority is separate from mining operation authority;
- claim boundaries are reviewed for public or investor-facing use;
- rollback owner and monitoring owner are assigned.
