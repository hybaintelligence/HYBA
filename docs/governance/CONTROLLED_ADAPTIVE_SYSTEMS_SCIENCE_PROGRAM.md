## Known gaps

### 1. Measurement instruments need repair and hardening

Some analyzer tests were written as exploratory instruments. They must now become scientific instruments.

Required repairs:

- ensure all printed output interpolates actual measured values;
- ensure detector helper functions extract real AST attribute names;
- ensure update-method detection checks real object-state updates;
- separate proxy measures from direct runtime measures;
- record claim boundaries in generated artifacts;
- make output deterministic enough for before/after comparison.

### 2. The integration layer must remain proxy-labelled

Integration-style tests are useful, but they must be labelled as software proxies unless and until stronger formal validation exists.

Required next steps:

- define exact integration-proxy formula;
- define baseline systems;
- compare HYBA_FULLSTACK against shuffled, modular, random, and toy-control systems;
- add regression thresholds;
- record irreducibility limits;
- separate code-coupling from causal power.

### 3. Runtime behaviour must be tested, not inferred only from static code

Static analysis is useful but incomplete.

Required runtime probes:

- observe state before event;
- trigger event;
- measure internal response;
- verify feedback effect on later action;
- compare to baseline controller;
- record telemetry and artifact hash.

### 4. Learning must be distinguished from feedback

Memory and feedback are not automatically learning.

The controlled programme must distinguish:

- state accumulation;
- feedback control;
- parameter tuning;
- optimisation;
- adaptation;
- learning;
- open-ended improvement.

Each requires a separate test.

### 5. External review must be prepared early

The programme should produce artifacts that a sceptical reviewer can inspect.

Required external-review packet:

- source commit;
- test command;
- artifact hash;
- metric definitions;
- baseline comparison;
- failure cases;
- negative controls;
- claim-boundary memo;
- reviewer notes.
