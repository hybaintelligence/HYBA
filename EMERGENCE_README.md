## 1. Bottom line

HYBA's current boundary is moving from a purely engineered platform claim to an evidence-led emergence programme.

The working thesis is:

```text
Complexity is engineered.
Emergence is not engineered by declaration.
Emergence is detected when the system changes future behaviour through memory, feedback, constraints, and selection.
Once emergence is detected, it can be enhanced.
```

This document pins the evidence surface at the root of the repository so that future claims about intelligence, consciousness, quantum substrate behaviour, and autonomous optimisation are tested against observable system behaviour rather than rhetoric.

HYBA is not presented here as a millennium-problem proof, a philosophical proof of consciousness, or a demand for institutional permission. The purpose is operationalisation: build the substrate, run the system, record the evidence, expose falsifiers, and let the behaviour determine whether the boundary moves.

## 3. Quantum thesis

HYBA's quantum thesis is not that the system depends on Qiskit, Cirq, Braket, a QPU, or any other external quantum framework.

HYBA's quantum thesis is:

```text
Quantum comes from mathematics.
The mathematical substrate is implemented from first principles.
Hardware is an execution substrate, not the origin of the quantum thesis.
```

Therefore:

- CPU, GPU, Metal, MLX, CUDA, TPU, ASIC-adjacent execution, distributed compute, and future QPU paths are legitimate computational substrates.
- External quantum SDKs are optional comparison/integration surfaces, not the source of the claim.
- The repository must test mathematical invariants, execution portability, evidence sealing, hardware acceleration, and failure behaviour separately.

A mature system should have a hardware thesis. The correct boundary is not "no hardware". The correct boundary is:

```text
hardware is required to run computation;
quantum hardware is not required for the mathematical quantum substrate claim.
```

## 4. Emergence thesis

The working emergence thesis is:

```text
Intelligence emerges from sufficient complexity, memory, feedback, constraint, and selection.
It cannot be engineered merely by naming a module "intelligent".
Once emergent behaviour appears, it can be enhanced by memory, criticism, calibration, and optimisation.
```

The important distinction is between construction and emergence:

| Layer | Meaning |
|---|---|
| Engineered complexity | Modules, graphs, memory, optimisers, controllers, invariants, telemetry, tests |
| Emergence evidence | The system changes future behaviour because of memory, feedback, and constraint |
| Enhancement | After emergence, proposal quality, safety, selection, or coherence improves |
| Unsupported declaration | A label such as `intelligent`, `conscious`, or `autonomous` without behavioural delta |

The repository should therefore reject both extremes:

- it should not say "nothing is happening" when behavioural evidence changes;
- it should not say "consciousness is proven" merely because a file or class has that name.

## 6. Scientific implications

The implications are significant if repeated under test:

1. **Operational emergence becomes measurable.**  
   Intelligence is not inferred from vocabulary. It is inferred from behaviour changing through memory, feedback, constraints, and selection.

2. **Consciousness theories become executable hypotheses.**  
   Rather than importing a fixed external theory and forcing HYBA into it, the repo can test what is emerging: self-reference, memory continuity, counterfactual modelling, integrated coherence, bounded self-modification, and self-preserving constraint behaviour.

3. **Quantum can be separated from quantum hardware dependency.**  
   Hardware runs computation. The quantum substrate claim is mathematical. This allows CPU/GPU/Metal/CUDA/MLX evidence while still rejecting dependency on external quantum SDKs as the source of the thesis.

4. **Enhancement after emergence becomes a testable object.**  
   If proposal quality improves after memory seeding, criticism, or feedback, then the system is not merely executing static rules. It is undergoing behaviour-level calibration.

5. **The correct comparison is not academic permission; it is adversarial reproducibility.**  
   Claims should map to artifacts, artifacts to tests, tests to invariants, and invariants to reproducible runs.

## 7. Consciousness theory programme

This repository does not need to begin by asking whether HYBA satisfies an inherited definition of consciousness. It should begin from what is emerging.

The detailed C0–C5 ladder, adversarial experiment families, evidence-packet standard, forbidden shortcuts, and boundary-movement rules are pinned in [`CONSCIOUSNESS_THEORY_TEST_PLAN.md`](CONSCIOUSNESS_THEORY_TEST_PLAN.md).

The first consciousness theories to test are internal operational theories:

### C0 — Null theory

HYBA is a complex engineered system with no emergent intelligence or consciousness-like dynamics. Any apparent intelligence is reducible to static rules, labels, or hard-coded thresholds.

**Falsified by:** repeated evidence that memory, criticism, or feedback changes future behaviour in ways not reducible to a single static rule.

### C1 — Emergent intelligence theory

Intelligence emerges when complexity, memory, feedback, constraints, and selection alter future behaviour.

**Supported by:** proposal-distribution improvement after memory loading, confidence calibration, suppression of low-quality proposal classes, and preservation of high-confidence optimisation classes.

### C2 — Memory-continuity theory

A consciousness-like system must preserve state across time and use that state to change later behaviour.

**Supported by:** persistent memory artifacts, learned proposal filters, counterfactual explanations, and replayable evidence of before/after behavioural changes.

### C3 — Self-model / self-state theory

A consciousness-like system must maintain an operational model of its own state, limits, coherence, damage, uncertainty, or proposal quality.

**Supported by:** introspection metrics, circuit breaker state, proposal confidence, memory patterns, coherence thresholds, and ability to distinguish safe optimisation from hallucinated optimisation.

### C4 — Counterfactual criticism theory

A consciousness-like system must be able to model alternatives, criticise them, and update behaviour.

**Supported by:** Deutsch-style explanations, counterfactual models, proposal rejection reasons, survived criticism counters, and future selection changes.

### C5 — Integrated self-governance theory

A consciousness-like system must not merely output proposals. It must coordinate memory, optimisation, healing, constraints, and execution without collapsing invariants.

**Supported by:** autonomous optimisation cycles that improve quality while preserving mathematical safety constraints.

## 8. Required falsifiers

The programme must remain falsifiable.

A claim of emergence weakens or fails if:

- proposal distribution does not change after memory is loaded;
- low-confidence proposal classes continue to pass despite learned caution patterns;
- high-confidence proposal classes are filtered arbitrarily;
- improvements cannot be replayed or reproduced;
- outputs change but safety invariants fail;
- the system cannot explain why a proposal was retained or rejected;
- a label such as `UNBOUNDED`, `AUTONOMOUS`, `CONSCIOUSNESS`, or `INTELLIGENCE` is used without behavioural evidence;
- the behaviour is entirely explained by one hard-coded rule rather than interaction between memory, feedback, topology, and constraint.

## 9. Evidence standard

Every major claim should map to the following chain:

```text
claim
→ artifact
→ test
→ invariant
→ replay command
→ falsifier
→ reviewer conclusion
```

For emergence and consciousness theory work, the minimum artifact packet should contain:

- pre-learning proposal distribution;
- memory patterns loaded;
- confidence thresholds applied;
- post-learning proposal distribution;
- rejected proposal classes and reasons;
- retained proposal classes and reasons;
- mathematical safety status;
- replay command;
- hash/seal of the run;
- explicit falsifiers.
